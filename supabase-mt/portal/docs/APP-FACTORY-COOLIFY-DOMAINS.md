# App Factory: Coolify API — Domains and custom subdomains

This doc surfaces the **official Coolify API** behaviour for setting application domains so we can debug cases where a custom subdomain is ignored and a random one is used.

## Official API reference (Context7 / docs)

- **Create (Public) application**  
  **POST** `/api/v1/applications/public`  
  https://coolify.io/docs/api-reference/api/operations/create-public-application  

  Request body (domain-related fields from the schema):

  | Field | Type | Description |
  |-------|------|--------------|
  | `domains` | **string** | Domain(s) for the application. Coolify UI/docs often show with scheme, e.g. `https://my-app.example.com`. For multiple domains, some sources use comma-separated (e.g. `https://a.com,https://b.com`). |
  | `autogenerate_domain` | **boolean** | When `true`, Coolify can auto-generate a subdomain (e.g. random subdomain under the server’s wildcard domain). When we send a custom domain we set this to **`false`**. |
  | `force_domain_override` | **boolean** | When `true`, our provided domain should override any server-level or auto-generated domain. **We set this to `true` when sending a custom domain** so Coolify uses our subdomain instead of generating one. |

- **Update application (PATCH)**  
  **PATCH** `/api/v1/applications/{uuid}`  
  Used to update `install_command`, `build_command`, `start_command`, and **`domains`**.  
  (Exact operations list: https://coolify.io/docs/api-reference/api — look for “Update” under Applications.)

  If the initial POST does not persist the domain, a follow-up PATCH with `domains` (and optionally redeploy) can be tried. Known issues (below) indicate PATCH domain updates sometimes do not take effect or do not refresh the proxy until redeploy.

## What the App Factory sends

- **On create** (`createCoolifyApplicationAndDeploy` in `portal/lib/app-factory/coolify.ts`):
  - When the deployment spec has a valid domain (or we derive `<slug>.<APP_FACTORY_ROOT_DOMAIN>`):
    - `domains`: `https://<domain>` (Coolify format with scheme).
    - `autogenerate_domain`: `false`.
    - `force_domain_override`: `true` (so our domain overrides any autogenerated one).
  - For **platforms** (Nixpacks) we then PATCH the same application with `install_command`, `build_command`, `start_command`, and again `domains`, then trigger deploy.

- **On “Fix Coolify”** (`fixCoolifyApplicationConfig`): we PATCH the application with optional `domains` (and commands), then trigger deploy.

## Known Coolify issues (GitHub)

- **Domains not set on create (API)**  
  https://github.com/coollabsio/coolify/issues/6915  
  - For **Docker Compose** apps you must use **`docker_compose_domains`** (array), not the string `domains`. Our App Factory uses **Nixpacks** for the platforms template, so we use the string `domains`.
  - Some versions had the “Domains” field remaining empty after creation; fixed in later betas (e.g. 435+); worth confirming your Coolify version.

- **PATCH to update domains not working as expected**  
  https://github.com/coollabsio/coolify/issues/4999  
  - PATCH domain updates can fail to persist or to refresh the proxy; redeploy or server config refresh may be needed.

- **Server config not refreshed after domain update via PATCH**  
  https://github.com/coollabsio/coolify/issues/5591  
  - After changing domains via API, the server/proxy might need a refresh for the new domain to be used.

## Why you might see the Coolify application UUID as the subdomain

Coolify’s **autogenerated** FQDN uses the **application UUID**, not the application name:

- With a server **wildcard domain** (e.g. `example.com`): `http://{uuid}.example.com` (e.g. `http://vgsco4o.example.com`).
- Without a wildcard: `http://{uuid}.{ip}.sslip.io`.

So if the custom `domains` value is not applied (e.g. API bug or wrong format), Coolify falls back to this UUID-based subdomain. The subdomain matching the “Coolify application” ID in the launch screen is this behaviour. Setting `autogenerate_domain: false` and `force_domain_override: true` plus sending `domains` should override it; we also **PATCH** the application with `domains` after create so the domain is set even if the initial POST drops it.

## Debugging checklist when custom subdomain is not used

1. **Coolify version**  
   Check your instance version. Some domain bugs were fixed in specific betas (e.g. 435+).  
   Downgrade/upgrade: https://coolify.io/docs/get-started/downgrade  

2. **Exact request body**  
   Log or inspect the POST body we send to `POST /api/v1/applications/public`: ensure `domains` is the full URL (e.g. `https://my-app.moodmnky.com`), `autogenerate_domain` is `false`, and `force_domain_override` is `true`.  

3. **Server wildcard domain**  
   If the Coolify **server** has a wildcard domain set, it may still assign a random subdomain unless we override it. Our code sets `force_domain_override: true` when we send a custom domain.  

4. **PATCH after create**  
   For platforms we PATCH the app with commands and `domains` before first deploy. If the POST dropped the domain, the PATCH might still set it; check Coolify UI “Domains” after create and after PATCH.  

5. **Env and redeploy**  
   After changing domain, we set `NEXT_PUBLIC_APP_URL` (and root domain) on the application and trigger deploy. Ensure the deployed app is restarted so it picks up the new env.  

6. **DNS**  
   Confirm the custom subdomain (e.g. `my-app.moodmnky.com`) resolves to the Coolify server (or proxy) and that wildcard DNS (`*.moodmnky.com`) is not required if you only use explicit subdomains.
